# lumin-chat 设计文档

## 1. 目标

本项目旨在构建一个名为 lumin-chat、在功能上接近 GitHub Copilot Terminal 的 Linux 终端代理：

- 自然语言任务输入
- 多轮代理循环
- 由 LLM 驱动的工具调用
- 带批准控制的 Shell 执行
- 支持黑名单/白名单命令策略的 Shell 执行
- 文件读取、写入、搜索与目录检查
- 可选的通过 SSH 的远程知识库查询
- 可恢复的会话
- 交互式聊天模式和一次性执行模式
- 可部署到 Linux 主机的最小化负载

它不是对 GitHub Copilot Terminal 的逐字节复制，因为官方产品的内部提示、策略、UI 协议、遥测和私有编排逻辑并未公开。这里的目标是在工作流程层面实现功能等价。

## 2. 产品形态

工具提供两种模式：

- `chat`：类 REPL 的交互式终端助手
- `ask`：用于脚本、CI 或远程测试的一次性执行

界面有意保持终端原生风格。仅在可读面板和流式输出处使用 Rich，而不是构建复杂的 TUI。

## 3. 核心能力

### 3.1 对话引擎

代理保持结构化的消息历史并将其持久化到磁盘，以便会话可恢复。

### 3.2 工具循环

LLM 可以发出工具调用。运行时执行这些调用，将结果追加回对话，并持续进行，直到模型不再请求工具或达到循环预算为止。

### 3.3 持久化 Shell

在 Linux 上，Shell 执行使用基于 PTY 的持久化 bash 会话。这一点很重要，因为它保留了：

- 跨命令的当前工作目录
- 跨命令导出的环境变量
- 更接近真实终端会话的 shell 状态

当运行时不在 Linux 上或显式提供 `cwd` 覆盖时，执行会回退到一次性子进程模式。

### 3.4 工具

当前内置工具：

- `run_shell_command`
- `change_directory`
- `list_directory`
- `search_text`
- `read_file`
- `write_file`
- `get_environment`
- `ssh_execute_command`
- `list_knowledge_documents`
- `read_knowledge_document`

这些覆盖了终端自动化、编码辅助、部署和诊断所需的最小功能集。

### 3.4.1 SSH 远程执行

新增 `ssh_execute_command` 工具，支持：

- 远程主机 IP 或域名
- SSH 端口
- 用户名
- 密码（也允许为空，交给密钥认证）
- 要执行的远端命令
- 超时时间与可选工作目录

返回结果会统一封装为退出码、标准输出、标准错误和目标主机信息，便于模型继续做自动化诊断。

### 3.5 审批策略

支持三种审批模式：

- `prompt`：在执行 shell 命令或写文件前询问
- `auto`：直接执行
- `read-only`：拒绝执行 shell 命令和写文件

这反映了类似 Copilot 风格终端工具的核心安全交互模式。

### 3.6 命令策略

Shell 执行有两种策略模式：

- `blacklist`：阻止包含禁止模式的命令，允许其余所有命令
- `whitelist`：仅允许顶层可执行名称被明确列出的命令

活动策略会注入到系统提示中，以便模型在一开始就避免建议被阻止的命令。

### 3.7 模型升级

当发生以下情况时，运行时可以自动将请求提升到更强的模型层级：

- LLM 调用失败
- 相同的工具调用连续重复多次
- 工具执行反复失败

这旨在减少在弱模型上出现的死循环，并匹配用户要求的基于等级的回退工作流。

### 3.8 知识库

可以通过 SSH 配置一个可选的远程文档库。代理可以：

- 先列出候选文档名称
- 按名称选择相关文档
- 仅读取所需的行范围

### 3.9 远端构建与部署

部署脚本支持两种模式：

- 本地打包后直接部署到测试板
- 先把源码上传到可选的构建服务器，在构建服务器执行 `python3 -m compileall` 与打包，再将产物部署到测试板

这使项目能够满足“远端构建 + 测试板验证”的完整交付流程。

这使终端代理能够整合组织或平台的背景资料，而无需将这些文档嵌入到项目中。

## 4. 运行时架构

### 4.1 模块

- `main.py`：轻量入口
- `src/app.py`：CLI 解析与聊天循环
- `src/agent.py`：多轮编排
- `src/ai_client.py`：LLM 传输与流解析
- `src/toolkit.py`：本地工具注册与执行
- `src/session_store.py`：会话持久化
- `src/config_loader.py`：配置加载与环境覆盖
- `src/prompts.py`：系统提示定义
- `src/ui.py`：流式与终端输出渲染
- `src/models.py`：共享数据类

### 4.2 消息流

1. 用户输入请求。
2. 代理将其追加到会话历史。
3. `AIClient` 发送完整的消息列表和工具定义。
4. 如果模型返回工具调用，代理执行这些调用。
5. 工具结果作为 `tool` 消息追加。
6. 代理继续调用模型，直到生成最终的助手响应。
7. 会话状态被保存。

## 5. 配置

`config.json` 包含模型定义、应用默认值和日志标志。

推荐的发展方向：

- 在生产环境中将密钥移到环境变量
- 保留多个模型层级以在成本/性能间权衡
- 添加按模型的回退规则

## 6. Linux 部署策略

部署使用一个小型暂存包，而不是上传完整工作区。

部署包中包含：

- 应用源代码
- 配置
- 依赖清单
- 文档
- 运行时脚本

有意排除：

- `.venv`
- `.dist` 构建产物
- `__pycache__`
- 本地编辑器元数据

## 7. 测试策略

### 7.1 本地

- 语法校验
- CLI 帮助验证
- 只读代理请求

### 7.2 远程 Linux

- 通过 `scripts/deploy.ps1` 上传
- 引导虚拟环境
- 安装依赖
- 运行 `python main.py --help`
- 在 `read-only` 模式下运行一次性提示
- 在 `prompt` 或 `auto` 模式下运行交互会话

## 8. 扩展路线图
