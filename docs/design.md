# lumin-chat 设计文档

## 1. 项目目标

lumin-chat 是一个运行在 Linux 终端中的中文友好型智能代理，目标是提供接近现代终端代理的工作流体验，并满足以下工程要求：

- 优先支持中文输入与输出，降低乱码风险。
- 交互模式下正常支持 Backspace、Delete、方向键和历史记录。
- 通过多轮工具调用完成代码、部署、测试和排障任务。
- 支持黑名单/白名单命令策略，避免生成危险指令。
- 支持模型分级与自动升模，降低弱模型反复失败的问题。
- 支持远程 SSH 操作、文档库读取、网页访问与网页搜索。
- 支持 RPM 打包、构建服务器编译、测试板安装与回归测试。

## 2. 运行模式

项目提供两种入口模式：

- `chat`：交互式终端会话。
- `ask`：单次请求模式，适合脚本和自动化调用。

交互模式使用 `prompt_toolkit` 读取输入，配合 UTF-8 标准流配置，提升中文输入、删除键、方向键和命令历史体验。

## 3. 核心能力

### 3.1 代理编排

`src/agent.py` 负责维护消息历史、驱动模型调用、执行工具以及处理自动升模逻辑。代理会把会话持续写入磁盘，便于恢复历史上下文。

### 3.2 Linux 持久 Shell

`src/toolkit.py` 内置基于 PTY 的持久 Bash 会话，具备以下特性：

- 记住当前工作目录。
- 记住通过 `export` 设置的环境变量。
- 更接近真实终端行为。
- 在 Linux 环境下优先复用同一个 shell，减少上下文丢失。

### 3.3 命令安全策略

系统同时支持：

- 黑名单模式：默认开启，仅阻止危险命令片段。
- 白名单模式：只允许显式批准的命令。
- 审批模式：`prompt`、`auto`、`read-only`。

黑白名单内容会注入系统提示词，帮助模型在生成命令前就规避风险。

### 3.4 模型自动升模

当出现以下情况时，系统会自动升级到更高 level：

- LLM 调用失败。
- 同一工具调用连续重复。
- 工具连续失败达到阈值。

这样可以减少死循环和反复生成错误命令的情况。

## 4. 工具体系

当前内置 20 个工具，分为 5 类。

### 4.1 本地终端与文件工具

- `run_shell_command`
- `change_directory`
- `list_directory`
- `search_text`
- `read_file`
- `write_file`
- `get_environment`

### 4.2 SSH 远程命令工具

- `ssh_execute_command`

### 4.3 SSH 远程文件管理工具

- `ssh_upload_file`
- `ssh_download_file`
- `ssh_list_directory`
- `ssh_read_file`
- `ssh_write_file`
- `ssh_make_directory`
- `ssh_remove_path`
- `ssh_path_exists`

这些工具通过 Paramiko/SFTP 提供能力，不依赖模型自行拼接复杂的远端 shell 操作，适合远程部署、日志收集和文件修复。

### 4.4 Web 工具

- `fetch_web_page`
- `search_web`

它们用于公开网页抓取、正文提取和搜索结果整理，适合查询在线资料、安装文档和技术说明。

### 4.5 远程文档库工具

- `list_knowledge_documents`
- `read_knowledge_document`

文档库通过 SSH/SFTP 访问，可先按文件名筛选，再按行读取具体内容，避免无差别读取大量文档。

## 5. 中文与终端交互改进

为满足中文终端使用体验，本项目做了以下处理：

- 标准输入、标准输出、标准错误统一重配置为 UTF-8。
- 交互输入改为 `prompt_toolkit`，解决普通 `input()` 在复杂终端下对中文输入和编辑按键支持不足的问题。
- Shell 返回内容统一按 UTF-8 解码，错误场景使用 `replace` 策略避免崩溃。
- 远端命令、网页和文档读取均尽量使用 UTF-8 输出。

## 6. 配置优先级

配置文件加载顺序如下：

1. 项目目录中的 `config.json`。
2. 如果存在 `/etc/lumin-chat/config.json`，则用它覆盖前者。
3. 之后再补齐默认值和环境变量覆盖。

这样可以保证安装后的系统配置优先生效，同时保留开发目录中的默认配置作为兜底。

## 7. RPM 打包与安装布局

RPM 安装后的目录约定如下：

- 程序源码：`/var/lib/lumin-chat`
- 系统配置：`/etc/lumin-chat/config.json`
- 启动脚本：`/usr/bin/lumin-chat`

RPM 安装阶段会自动尝试完成以下动作：

- 创建项目虚拟环境。
- 安装 `requirements.txt` 依赖。
- 在无法创建虚拟环境时回退到 `vendor` 目录安装依赖。
- 编译 Python 字节码，减少首次运行检查成本。

## 8. 构建服务器与部署流程

`deploy.py` 支持两种打包格式：

- `rpm`：默认模式，先构建 RPM，再上传到目标机执行安装。
- `source`：保留源码部署模式，适合临时调试。

同时支持两种构建路径：

- 本地构建后直接部署。
- 先把最小源码包上传到远端构建服务器，在构建服务器执行 `compileall` 和 `build_rpm.py`，再把产物部署到测试板。

如果目标主机使用 SSH alias，例如 `tl3588`，脚本也支持在未显式指定用户名时沿用 alias 中的用户配置。

## 9. 测试策略

### 9.1 本地测试

- `scripts/smoke_test.py`
- `python3 -m compileall main.py src scripts`
- `lumin-chat --help` 等基础帮助验证

### 9.2 远端测试

部署完成后，`deploy.py --run-tests` 会自动执行：

- RPM 安装状态检查
- 系统配置文件检查
- 启动脚本帮助信息检查
- 项目冒烟测试
- Docker Ubuntu 非交互测试
- Docker 容器内文件写入与目录遍历验证

最终会生成中文 Markdown 报告，记录测试命令、输出结果与汇总结论。

## 10. 代码整洁原则

项目保持以下约束：

- 只保留一份中文设计文档。
- 删除重复或无必要的文档文件。
- 代码尽量使用模块化辅助函数，避免在入口脚本中堆积逻辑。
- 所有新增模块、类和方法都补充中文说明性注释或文档字符串。
