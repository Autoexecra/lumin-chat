# lumin-chat 设计文档

## 1. 目标

本项目旨在构建一个名为 lumin-chat 的 Linux 终端代理，在工作流层面尽量接近 GitHub Copilot Terminal：

- 自然语言任务输入
- 多轮代理循环
- 由 LLM 驱动的工具调用
- 带审批控制的 Shell 执行
- 支持黑名单/白名单命令策略的 Shell 执行
- 文件读取、写入、搜索与目录检查
- 可选的基于 SSH 的远程知识库检索
- 可恢复会话
- 交互式聊天模式与一次性 ask 模式
- 可用最小负载部署到 Linux 主机

它不是对 GitHub Copilot Terminal 的逐字节复刻，因为官方产品的内部提示词、策略、UI 协议、遥测与私有编排逻辑并未公开。这里的目标是在工作流层面实现尽可能接近的功能体验。

## 2. 产品形态

工具提供两种运行模式：

- `chat`：类 REPL 的交互式终端助手
- `ask`：用于脚本、CI 或远程测试的一次性执行

界面保持终端原生风格。Rich 仅用于更清晰的面板和流式输出展示，不引入复杂 TUI。

## 3. 核心能力

### 3.1 对话引擎

代理会维护结构化消息历史，并持久化到磁盘，以便会话恢复。

### 3.2 工具循环

LLM 可以发起工具调用。运行时执行工具、把结果写回上下文，并持续循环，直到模型不再请求工具或达到轮次上限。

### 3.3 持久 Shell

在 Linux 上，Shell 执行基于 PTY 维护持久 bash 会话。这样做的意义是可以保留：

- 跨命令保持当前工作目录
- 跨命令保持导出的环境变量
- 更接近真实终端的 shell 状态

当运行环境不是 Linux，或者显式指定了 `cwd` 覆盖值时，会回退到一次性子进程模式。

### 3.4 工具集合

当前内置工具包括：

- `run_shell_command`
- `change_directory`
- `list_directory`
- `search_text`
- `read_file`
- `write_file`
- `get_environment`
- `ssh_execute_command`
- `list_knowledge_documents`
- `read_knowledge_document`

这些工具覆盖了终端自动化、编码辅助、部署与诊断所需的最小能力集合。

### 3.4.1 远程 SSH 执行

新增的 `ssh_execute_command` 工具支持以下参数：

- 远程主机 IP 或域名
- 端口号
- 用户名
- 密码（可为空，表示优先使用密钥登录）
- 要执行的命令
- 超时时间与可选远端工作目录

工具返回结构化结果，包含退出码、标准输出、标准错误与连接目标信息，便于模型基于结果继续诊断。

### 3.5 审批策略

当前支持三种审批模式：

- `prompt`：执行 shell 或写文件前先确认
- `auto`：直接执行
- `read-only`：拒绝 shell 执行和文件写入

这对应了 Copilot 类终端工具最核心的安全交互方式。

### 3.6 命令策略

Shell 执行支持两种策略模式：

- `blacklist`：拦截包含危险模式的命令，其余默认放行
- `whitelist`：仅允许顶层可执行名称出现在白名单中的命令

当前策略会写入 system prompt，尽量让模型从源头避免生成被策略阻止的命令。

### 3.7 自动升模

在以下情况出现时，运行时可以自动升级到更高模型层级：

- LLM 调用失败
- 同一工具调用连续重复多次
- 工具执行连续失败

这样可以减少弱模型上的死循环，并满足按 level 逐级兜底的要求。

### 3.8 知识库

系统可配置基于 SSH 的远程文档库。代理可以：

- 先列出候选文档名
- 再按名称选择相关文档
- 最后只读取需要的行范围

### 3.9 远端构建与部署

部署脚本支持两条路径：

- 本地直接打包并部署到测试板
- 先上传到可选的构建服务器，执行 `python3 -m compileall` 校验并产出压缩包，再下发到测试板

这样可以满足“先在编译服务器处理，再部署到测试板验证”的流程要求。

这样就能在不把大量板卡资料直接打包进工程的前提下，引入平台背景知识。

## 4. 运行时架构

### 4.1 模块划分

- `main.py`：轻量入口
- `src/app.py`：CLI 解析与聊天循环
- `src/agent.py`：多轮代理编排
- `src/ai_client.py`：LLM 传输与流式解析
- `src/toolkit.py`：本地工具注册与执行
- `src/session_store.py`：会话持久化
- `src/config_loader.py`：配置加载与环境变量覆盖
- `src/prompts.py`：系统提示词构造
- `src/ui.py`：终端渲染
- `src/models.py`：共享数据结构

### 4.2 消息流

1. 用户输入请求。
2. 代理把请求追加到会话历史。
3. `AIClient` 发送完整消息列表与工具定义。
4. 如果模型返回工具调用，代理执行工具。
5. 工具结果以 `tool` 消息回填上下文。
6. 代理继续调用模型，直到产出最终回答。
7. 会话状态落盘保存。

## 5. 配置

`config.json` 包含模型定义、应用默认值、命令策略、知识库与部署配置。

推荐的后续演进方向：

- 生产环境将密钥迁移到环境变量
- 保留多层模型以平衡成本和效果
- 增加更细粒度的 per-model 回退规则

## 6. Linux 部署策略

部署过程只上传精简后的暂存包，而不是整个工作区。

部署包中包含：

- 应用源码
- 配置文件
- 依赖清单
- 文档
- 运行脚本

有意排除：

- `.venv`
- `.dist` 构建产物
- `__pycache__`
- 本地编辑器元数据

## 7. 测试策略

### 7.1 本地测试

- 语法校验
- CLI 帮助检查
- 只读模式代理调用

### 7.2 远端 Linux 测试

- 通过 `scripts/deploy.ps1` 或 `deploy.py` 上传
- 初始化虚拟环境
- 安装依赖
- 运行 `python main.py --help`
- 运行一次 ask 模式验证
- 在 `auto` 或 `prompt` 模式下执行交互验证

## 8. 扩展路线图

如果后续要继续逼近 GitHub Copilot Terminal 的体验，可以扩展：

- 面向代码修改的补丁工具，而不是整文件改写
- 写入前更丰富的 diff 展示
- 命令风险分级
- 会话摘要与压缩
- shell 环境变量更稳定的持久化
- 远程 MCP 风格工具插件
- 更完整的 slash command 体系

## 9. 后续 AI 编码任务建议

这个设计文档可直接用于后续 AI 编码任务，例如：

- 为 Linux 持久 PTY 增加更完整的异常恢复
- 添加基于补丁的文件编辑工具
- 增加命令 allowlist 或沙箱执行能力
- 为工具执行器补测试集
- 增加 Markdown 会话导出
- 增加配置校验命令
